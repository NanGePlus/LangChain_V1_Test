# 16_MultiAgentAPIServerLangGraphSupervisor 相对 14 的主要调整说明

相对 **14_AgentAPIServerWithPlaywright**，本项目将**单 Agent 全能架构**演进为官方 `langgraph-supervisor` 多智能体 Supervisor 架构，核心变化如下。

---

## 架构对比


| 维度                   | EP14（单 Agent）                                                    | 本 EP（langgraph_supervisor）                                                                            |
| -------------------- | ---------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Agent 数量             | 1 个全能 Agent                                                      | 1 个 Supervisor + 3 个 Sub-Agent（Web / Knowledge / Task）                                                |
| 工具注册入口               | 单一 `get_tools()` 汇总全部工具                                          | `get_web_agent_tools()` / `get_knowledge_agent_tools()` / `get_task_agent_tools()` 分域组装               |
| Supervisor           | 无                                                                | `create_supervisor(agents=[...], model=..., prompt=..., output_mode=...)`                             |
| Sub-Agent 调用方式       | —                                                                | Sub-Agent 为 **StateGraph 节点**，通过 `Command(goto, graph=Command.PARENT)` 图级路由                           |
| Handoff              | 无                                                                | 框架生成 `transfer_to_<agent_name>` 类工具                                                                   |
| checkpointer / store | 每请求新建 `create_agent` 图实例，但 **复用** 全局同一套 `checkpointer` 与 `store` | 仅 `compiled_supervisor` 在 `compile` 时绑定 `checkpointer`/`store`；Sub-Agent 子图不单独带 checkpointer，由父图统一持久化 |
| `user_id`            | `context=Context(user_id=...)` + `context_schema=Context`        | 主路径通过 `config["configurable"]["user_id"]` 透传；工具内 `get_config()` 读取                                    |
| Agent 构建             | 每请求 `create_agent_instance()`                                    | `lifespan` 内构建 Sub-Agent + **一次性** `compile` Supervisor，全进程复用                                         |
| API 端口               | `8203`                                                           | `8204`（与 EP14 区分，避免同机并行冲突）                                                                            |

## 1. 新增目录：`agents/`

EP14 **没有**独立的 Agent 模块，所有逻辑集中在 `agent_api.py` + `utils/tools.py` 的 `get_tools()`；本 EP 新增：


| 文件                          | 职责                               |
| --------------------------- | -------------------------------- |
| `agents/web_agent.py`       | Web Agent：Playwright 工具 + HITL   |
| `agents/knowledge_agent.py` | Knowledge Agent：MCP RAG + HITL   |
| `agents/task_agent.py`      | Task Agent：天气 / 位置 / 技能执行 + HITL |


三个 Sub-Agent 在构建时均需设置 `name`（如 `name="web_agent"`）：`create_supervisor` 据此注册图节点并生成对应的 **transfer** 工具名。

## 4. `agent_api.py`（编排层核心差异）


| 变化点       | EP14                                 | 本 EP                                                          |
| --------- | ------------------------------------ | ------------------------------------------------------------- |
| 创建方式      | `create_agent` + 中间件（如摘要）            | `create_supervisor(...).compile(checkpointer=..., store=...)` |
| 请求内实例     | 每次请求 `await create_agent_instance()` | 使用全局 `compiled_supervisor`                                    |
| HITL 中断归属 | 单 Agent 图内                           | Sub-Agent 内触发，interrupt **冒泡**到 Supervisor 编译图                |
| 接口形态      | 四路由                                  | **与 EP14 一致**（无破坏性变更）                                         |


`lifespan` 中编译 Supervisor 的要点（示意）：

```python
from langgraph_supervisor import create_supervisor

supervisor_graph = create_supervisor(
    agents=[web_agent, knowledge_agent, task_agent],
    model=llm_chat,
    prompt=supervisor_prompt,
    output_mode=Config.SUPERVISOR_OUTPUT_MODE,
)

compiled_supervisor = supervisor_graph.compile(
    checkpointer=checkpointer,
    store=store,
)
```

### 6.1 HITL 决策协议（与 EP14 一致建议）

`interrupt_details.action_requests` **不保证**含 `id`，`decisions` 宜使用：

- `{"type": "approve"}` / `{"type": "reject"}`
- `{"type": "edit", "edited_action": {"name": "...", "args": {...}}}`

### 6.2 终端交互式审批与多轮恢复

与 EP14 相同：支持 approve / reject / edit N / quit，以及 `/ask` → interrupted → `/intervene` → … 多轮循环。

### 6.3 SSE 流式表现

与 EP14 一致：状态码、token 分段、`tool_output` 块、`--debug` 等约定保持同一测试体验。

## 7. 多中断（multiple pending interrupts）与 `interrupt_id`

多 Sub-Agent / 多工具中断时，LangGraph 要求恢复必须带 `interrupt_id`：

> When there are multiple pending interrupts, you must specify the interrupt id when resuming.

本 EP 约定：

1. **服务端**：`status=interrupted` 时，在 `interrupt_details` 中附带 `interrupt_id`（若运行时有提供）。
2. **客户端**：`/intervene`、`/intervene/stream` 的请求体（`InterveneRequest`）中携带 `interrupt_id` 字段，与 EP14 相比为增量字段（EP14 仅 `thread_id`、`user_id`、`decisions`）。

### 7.1 排坑：`httpx` 与代理

部分环境下 `HTTP_PROXY`/`HTTPS_PROXY` 会导致访问 `localhost` 走代理出现 **502**；本 EP 测试脚本在流式恢复等路径可使用 `httpx.Client(..., trust_env=False)` 强制直连（与 EP14 纯 `requests` 场景相比更需注意本地代理变量）。
