# 15_MultiAgentAPIServerAgentsAsTools 相对 14 的主要调整说明

相对 **14_AgentAPIServerWithPlaywright**，本项目将**单 Agent 全能架构**演进为官方 LangChain 文档([subagents-personal-assistant](https://docs.langchain.com/oss/python/langchain/multi-agent/subagents-personal-assistant))推荐的 **Agents as Tools** 多智能体架构，核心变化如下。

---

## 架构对比


| 维度               | EP14（单 Agent）                       | 本 EP（Agents as Tools）                                        |
| ---------------- | ----------------------------------- | ------------------------------------------------------------ |
| Agent 数量         | 1 个全能 Agent                         | 1 个 Supervisor + 3 个 Sub-Agent                               |
| 工具分布             | 全部工具挂在单一 Agent                      | 按领域分组：Web / Knowledge / Task                                 |
| Supervisor 创建    | 无                                   | `create_agent(tools=[sub_agent_tools...], checkpointer=...)` |
| Sub-Agent 调用     | —                                   | `@tool` 包装器内 `sub_agent.ainvoke(request)`                    |
| Agent 实例化        | 每请求调用 `create_agent_instance()`     | lifespan 中一次性构建，全程复用                                         |
| `user_id` 传递     | `context=Context(user_id=...)` 显式传入 | `get_config()` 从 configurable 读取，工具层透传                       |
| checkpointer 归属  | Agent 本身持有                          | 只有 Supervisor 持有（Sub-Agent 无状态）                              |
| `context_schema` | `context_schema=Context`            | 已移除                                                          |
| 端口               | 8203                                | 8203（不变）                                                     |

## 1. 新增目录：`agents/`

EP14 没有独立的 Agent 模块；本 EP 新增：


| 文件                          | 职责                                    |
| --------------------------- | ------------------------------------- |
| `agents/web_agent.py`       | 构建 Web Agent（Playwright 浏览器工具 + HITL） |
| `agents/knowledge_agent.py` | 构建 Knowledge Agent（MCP RAG 检索 + HITL） |
| `agents/task_agent.py`      | 构建 Task Agent（天气/位置/技能执行 + HITL）      |


三个 Sub-Agent 均无 `checkpointer`，状态由 Supervisor 统一管理。

## 4. `agent_api.py`（最核心变化）


| 变化点           | EP14                                                        | 本 EP                                                                |
| ------------- | ----------------------------------------------------------- | ------------------------------------------------------------------- |
| Agent 构建时机    | 每请求调用 `create_agent_instance()`                             | lifespan 中一次性构建，全程复用                                                |
| Supervisor 构建 | 无                                                           | `create_agent(tools=supervisor_tools, checkpointer=..., store=...)` |
| Sub-Agent 传参  | —                                                           | 三个 Sub-Agent 作为参数传入 `get_supervisor_tools()`                        |
| context 传递    | `agent.ainvoke(..., context=Context(user_id=...))`          | 移除 `context=` 参数，改在 `config["configurable"]["user_id"]` 中透传         |
| 新增辅助函数        | —                                                           | `_extract_hitl_details()` 统一处理 Sub-Agent HITL 中断格式                  |
| HITL 中断来源     | 单 Agent 工具直接触发                                              | Sub-Agent 内部触发，经 `@tool` 包装器向上传播至 Supervisor                        |
| 接口形态          | `/ask` / `/ask/stream` / `/intervene` / `/intervene/stream` | **完全一致**，无破坏性变更                                                     |


**lifespan 构建 Supervisor 的关键代码：**

```python
# Sub-Agent 在 lifespan 启动时构建一次
web_agent    = await build_web_agent(llm_chat)
knowledge_agent = await build_knowledge_agent(llm_chat)
task_agent   = await build_task_agent(llm_chat)

# 将 Sub-Agent 包装为 Supervisor 工具
supervisor_tools = get_supervisor_tools(web_agent, knowledge_agent, task_agent)

# Supervisor 持有 checkpointer 和 store（Sub-Agent 无状态）
supervisor_agent = create_agent(
    model=llm_chat,
    system_prompt=supervisor_prompt,
    tools=supervisor_tools,
    checkpointer=checkpointer,
    store=store,
)
```

**与 EP14 `create_agent_instance()` 的对比：**

```python
# EP14：每请求创建 Agent，Agent 自身持有 checkpointer
async def create_agent_instance():
    agent = create_agent(
        model=llm_chat,
        tools=tools,                    # 全部工具混合
        context_schema=Context,         # 显式 Context 传参
        checkpointer=checkpointer,      # Agent 持有
        store=store
    )
    return agent

# 本 EP：lifespan 中一次构建，Supervisor 持有 checkpointer
supervisor_agent = create_agent(
    model=llm_chat,
    tools=supervisor_tools,             # 仅 Sub-Agent 包装工具
    # 无 context_schema
    checkpointer=checkpointer,          # 只有 Supervisor 持有
    store=store
)
```

| 变化                 | 说明                                                       |
| ------------------ | -------------------------------------------------------- |

### 6.1 与 EP14 一致的“终端交互式”审批体验

为保持与 EP14 的交互方式一致，EP15 的测试脚本在遇到 HITL 中断时用户在终端中选择：

- `approve`：全部通过
- `reject`：全部拒绝
- `edit N`：编辑第 N 个工具调用参数（其余默认 approve）
- `quit`：退出测试

并支持**多轮 HITL**：`/ask`→（interrupted）→交互审核→`/intervene`→（可能再次 interrupted）→… 直到 completed

### 6.2 与 EP14 一致的 SSE 流式输出效果

为对齐 EP14 的流式输出体验，EP15 的 `api_test.py` / `api_test_plus.py` 在流式模式下遵循同样的输出格式：

- 打印状态码
- token 逐段输出
- `tool_output` 以块形式打印：
  - `[工具输出] ... [/工具输出]`
- `--debug` 时在 stderr 打印 token 序号与长度
- 支持 `completed` / `interrupted` 的多轮循环

## 7. 多中断（multiple pending interrupts）恢复机制：新增 `interrupt_id`

在多 Agent 场景中，一个线程可能同时存在多个 pending interrupts。
此时如果仅用 `Command(resume={"decisions": ...})` 恢复，会触发 LangGraph 运行时错误：

> When there are multiple pending interrupts, you must specify the interrupt id when resuming.

为解决该问题，本 EP 做了两处同步增强：

### 7.1 服务端（`agent_api.py`）返回 `interrupt_id`

当返回 `interrupted` 时，服务端会在 `interrupt_details` 中附带：

- `interrupt_details.interrupt_id`

用于标识本次需要恢复的具体 interrupt（task id）。

### 7.2 客户端（`api_test.py`）提交 `interrupt_id`

客户端在调用 `/intervene` 与 `/intervene/stream` 时会把 `interrupt_id` 原样带回：

- `{"thread_id": "...", "user_id": "...", "interrupt_id": "...", "decisions": [...] }`

服务端在恢复时会按是否存在 `interrupt_id` 构造两种 resume 形态：

- 单中断：`Command(resume={"decisions": decisions})`
- 多中断：`Command(resume={interrupt_id: {"decisions": decisions}})`

### 7.3 接口兼容性说明（轻微增强）

接口路径保持与 EP14 一致（`/ask` / `/ask/stream` / `/intervene` / `/intervene/stream`），但本 EP 的 `/intervene` 与 `/intervene/stream` **请求体新增可选字段**：

- `interrupt_id?: string`

不传则保持原行为；当出现多 pending interrupts 时建议传入以保证可恢复执行
